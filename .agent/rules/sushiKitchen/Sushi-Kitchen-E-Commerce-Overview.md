# Sushi Kitchen’s E-Commerce Overview

Sushi Kitchen’s e-commerce CMS is Payload CMS, hosted as the payload service and wired up via payload/payload.config.ts where we bootstrap the API key Medusa uses to talk to Payload (payload/payload.config.ts (lines 15-140)). Payload runs as a headless editor surface for marketing/content teams while Medusa remains the transactional catalog/checkout engine.

Medusa drives all synchronization into Payload through dedicated workflows and event subscribers. When a product is created/updated/deleted, the Medusa subscribers fire (medusa/src/subscribers/product-created.js (line 1), .../product-deleted.js, .../variant-updated.js (line 1)), which invoke workflows like create-payload-products (medusa/src/workflows/create-payload-products.js (lines 12-84)). 

Those workflows pull the authoritative product data (title, subtitle, description, options, variants, media) from Medusa’s graph query and POST it into Payload’s products collection via the shared PayloadModuleService (medusa/src/modules/payload/service.js (lines 1-168)). After each sync, the Medusa product metadata is updated with the Payload document id so future edits can round-trip cleanly (medusa/src/workflows/create-payload-products.js (lines 66-78)). Similar workflows exist for options and variants (medusa/src/workflows/create-payload-product-options.js (lines 10-71), medusa/src/workflows/update-payload-product-variants.js, etc.), ensuring that any structural changes made in Medusa immediately cascade to the CMS copy.

On the Payload side, the products collection is intentionally shaped for human-friendly editing while locking down the fields that must stay in sync with Medusa (payload/src/collections/products.ts (lines 5-284)). The helper isMedusaRequest flag (payload/src/collections/products.ts (line 3)) allows only requests tagged ?is\_from\_medusa=true (the Medusa workflows set this flag at the SDK layer) to create/delete documents or touch IDs and inventory-meaningful arrays. Editors can freely update marketing copy—rich text descriptions, subtitles, SEO metadata, hero media—because those fields remain editable in the admin UI (payload/src/collections/products.ts (lines 27-122)). 

Conversely, validation hooks block humans from adding/removing options or variants by comparing the incoming edit against the previously synced arrays and rejecting mismatches (payload/src/collections/products.ts (lines 124-270)). This keeps dimensions, variant matrices, and option ids authoritative in Medusa while still letting editors adjust phrasing, storytelling, and imagery.

Operationally, this split means content teams work entirely inside Payload’s admin screen (cms.sushikitchen.io/admin), benefiting from its Lexical rich text editor and media handling, while the storefront/Medusa API queries both systems: Medusa for pricing/inventory and Payload for the long-form product narratives or bundle landing sections (complete-network-topology.md (lines 49-495), docs/implementation/rewards/rewards-implementation-guide.md (lines 45-191)). Maintenance stays simple—marketing doesn’t need access to Medusa, and engineers don’t worry about manual edits breaking product schemas because only Medusa-signed requests can mutate those schema-critical fields.
